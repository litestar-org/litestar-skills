# Wheel Build + Asset Bundling

How to produce a single `.whl` that contains Python code **and** a compiled frontend — with no separate CDN, no runtime `bun run build`, and no post-install steps.

## The two patterns

### Pattern A — `force-include` (explicit)

Used by **[litestar-fullstack-inertia](https://github.com/litestar-org/litestar-fullstack-inertia)**. The built asset directory is `.gitignore`d; Hatchling would normally exclude it. `[tool.hatch.build.targets.wheel.force-include]` overrides that per-directory.

```toml
# pyproject.toml (litestar-fullstack-inertia)
[build-system]
build-backend = "hatchling.build"
requires = ["hatchling"]

[tool.hatch.metadata]
allow-direct-references = true

[tool.hatch.build.targets.sdist]
exclude = [".github", "docs"]
include = [
  "CHANGELOG.md",
  "app/domain/web/public/**/*",    # sdist also includes built output
  "app/**/*.sql",
  "app/db/**/*.ini",
  "tests",
  "resources",                      # raw React source for reproducibility
  "tools/deploy",
  "docs",
]
packages = ["app"]
skip-excluded-dirs = false

[tool.hatch.build.targets.wheel]
include = ["CHANGELOG.md", "/app/**/*.sql", "/app/db/**/*.ini"]
packages = ["app"]

# Force include gitignored built assets — generated by `uv run app assets build`
[tool.hatch.build.targets.wheel.force-include]
"app/domain/web/public" = "app/domain/web/public"

[tool.hatch.build.targets.binary]
pyapp-version = "0.29.0"
python-version = "3.12"
scripts = ["app"]
```

**When to use:**

- You want `.gitignore` to stay the source of truth for what's in VCS.
- You want an explicit, auditable list of what the wheel contains.
- You have multiple built-asset directories and want per-directory control.

### Pattern B — `ignore-vcs = true` (implicit)

Used by **[litestar-fullstack](https://github.com/litestar-org/litestar-fullstack)**. Hatchling ignores `.gitignore`; every file under the `packages = [...]` trees ships.

```toml
# pyproject.toml (litestar-fullstack)
[build-system]
build-backend = "hatchling.build"
requires = ["hatchling", "setuptools"]

[tool.hatch.metadata]
allow-direct-references = true

[tool.hatch.build]
dev-mode-dirs = ["src/py", "."]
ignore-vcs = true                        # ← the whole package tree, not just tracked files
sources = ["src/py"]

[tool.hatch.build.targets.sdist]
exclude = [".github", "docs"]
include = ["src/**/*", "tools/*", "resources", "LICENSE"]
skip-excluded-dirs = false

[tool.hatch.build.targets.wheel]
packages = ["src/py/app"]
# ignore-vcs is inherited from [tool.hatch.build]
```

A more elaborate variant uses the same core idea but spells out artifact patterns:

```toml
# pyproject.toml
[build-system]
build-backend = "hatchling.build"
requires = ["hatchling", "setuptools"]

[tool.hatch.metadata]
allow-direct-references = true

[tool.hatch.build]
dev-mode-dirs = ["src/py", "."]
ignore-vcs = true

[tool.hatch.build.targets.wheel]
packages = ["src/py/<app>"]
artifacts = [
  "**/*.j2", "**/*.sql", "**/*.ini",
  "**/*.ico", "**/*.png", "**/*.svg",
  "**/*.css", "**/*.js", "**/*.html",
  "**/*.txt", "**/*.json",
]

[tool.hatch.build.targets.wheel.sources]
"src/py" = ""

[tool.hatch.build.targets.binary]
scripts = ["<app>"]
pyapp-version = "0.29.0"
python-version = "3.13"

[tool.hatch.build.targets.binary.env]
PYAPP_DISTRIBUTION_EMBED = "1"
PYAPP_FULL_ISOLATION = "1"
PYAPP_UV_ENABLED = "1"
PYAPP_ALLOW_UPDATES = "1"
```

**When to use:**

- You want one switch to "include everything the package tree contains".
- You trust your dev workflow not to leave stray files under `src/py/<pkg>/`.
- You prefer one rule over per-directory `force-include` entries.

## Where Vite writes (and why it matters)

The wheel config above only works because **the frontend build writes INTO the Python package directory**. This is a deliberate choice, encoded in `vite.config.ts`:

```ts
// litestar-fullstack/src/js/web/vite.config.ts
import path from "node:path"
import litestar from "litestar-vite-plugin"
import { defineConfig } from "vite"

export default defineConfig({
  base: process.env.ASSET_URL ?? "/static/web/",
  build: {
    outDir: path.resolve(__dirname, "../../py/app/server/static/web"),   // ← inside Python package
    emptyOutDir: true,
  },
  plugins: [
    litestar({
      input: ["src/main.tsx", "src/styles.css"],
      bundleDir: path.resolve(__dirname, "../../py/app/server/static/web"),
      hotFile: path.resolve(__dirname, "../../py/app/server/static/web/hot"),
    }),
  ],
})
```

`outDir` is an absolute path into `src/py/app/server/static/web/`. Because `src/py/app` is the Hatchling package, anything under `static/web/` ships.

For litestar-fullstack-inertia, the same effect is achieved by configuring litestar-vite's `bundle_dir` (in the Python settings module) rather than Vite's `outDir`. The litestar-vite plugin then writes there when `app assets build` is invoked:

```python
# app/lib/settings.py
return ViteConfig(
    dev_mode=self.DEV_MODE,
    runtime=RuntimeConfig(executor="bun", trusted_proxies="*"),
    paths=PathConfig(
        root=BASE_DIR.parent,
        bundle_dir=Path("app/domain/web/public"),   # ← inside `app/` package
        resource_dir=Path("resources"),
    ),
)
```

**If you remember nothing else:** `bundle_dir` (and Vite `outDir`) must be a path **inside** a directory listed in `[tool.hatch.build.targets.wheel] packages = [...]`. A Vite default of `./dist` at the repo root produces an empty wheel.

## The Makefile chain

### Minimal (inertia-style)

```makefile
# https://github.com/litestar-org/litestar-fullstack-inertia — Makefile

.PHONY: install
install:
	@uv sync --all-extras --dev
	@bun install --frozen-lockfile

.PHONY: build-assets
build-assets:
	@uv run app assets build

.PHONY: build-wheel
build-wheel: build-assets
	@uv build --wheel

.PHONY: build-binary
build-binary: build-wheel
	@uv run hatch build --target binary
```

### Multi-asset

```makefile
# Makefile

.PHONY: js-install
js-install:
	@cd src/js/web && bun install --frozen-lockfile
	@cd src/js/templates && bun install --frozen-lockfile

.PHONY: js-build-web
js-build-web: js-install
	@uv run python manage.py assets build          # writes to src/py/<app>/server/public/

.PHONY: js-build-offline-report
js-build-offline-report: js-install
	@cd src/js/offline && bun run build            # writes to src/py/<app>/domain/web/static/reports/offline/

.PHONY: js-build-all
js-build-all: js-build-web js-build-offline-report

.PHONY: build-templates
build-templates:
	@cd src/js/templates && bun run build          # writes to src/py/<app>/domain/notifications/email/templates/

.PHONY: generate-licenses
generate-licenses:
	@uv run python tools/scripts/collect_python_licenses.py
	@uv run python tools/scripts/collect_js_licenses.py
	@uv run python tools/scripts/generate_sbom.py

.PHONY: build-wheel
build-wheel: generate-licenses build-templates js-build-all
	@uv build --wheel
```

Every prerequisite writes files **into the Python package directory**. `uv build --wheel` then captures them all in one pass.

## Verifying the wheel

After `make build-wheel`:

```bash
# List everything in the wheel
unzip -l dist/*.whl

# Check the frontend made it in
unzip -l dist/*.whl | grep -E '\.(js|css|html|svg|png)$' | head

# Check SQL / templates / licenses
unzip -l dist/*.whl | grep -E '\.(sql|j2|ini)$' | head

# Install in an empty env and test it starts
uv venv /tmp/test-venv
uv pip install --python /tmp/test-venv/bin/python dist/*.whl
/tmp/test-venv/bin/python -c "import app; print(app.__file__)"
```

If the `grep` for frontend extensions is empty, one of three things is wrong:

1. Vite `outDir` is pointing outside the Python package → fix `vite.config.ts`.
2. `[tool.hatch.build.targets.wheel]` doesn't include the built directory → switch to `force-include` or `ignore-vcs = true`.
3. `bun run build` wasn't run before `uv build` → fix Makefile dependency chain.

## Common mistakes

### Mistake 1: Vite writes to `./dist` at the repo root

**Symptom:** Wheel has `.py` files but no `.js`/`.css`.

**Fix:** Set `build.outDir` to an absolute path inside `src/py/<pkg>/` (or `<pkg>/`):

```ts
import path from "node:path"
build: {
  outDir: path.resolve(__dirname, "../../py/app/server/static/web"),
  emptyOutDir: true,
},
```

### Mistake 2: Using `force-include` AND `ignore-vcs = true`

**Symptom:** Duplicate-file warnings from Hatchling, flaky wheel contents.

**Fix:** Pick one. Default to `ignore-vcs = true` for new projects; use `force-include` when you need per-directory control.

### Mistake 3: Running `uv build` before building assets

**Symptom:** Wheel is tiny (~1 MB), frontend is missing, everything else looks fine.

**Fix:** Add `build-assets` as a Makefile prerequisite of `build-wheel`. In CI, put the asset-build step before `uv build`.

### Mistake 4: No placeholder directory in CI lint/typecheck jobs

**Symptom:** `mypy`, `pyright`, or `pre-commit` jobs fail with `hatchling` errors about missing paths even though they don't care about assets.

**Fix:** Every CI job that runs `uv sync` (not just ones that build) needs:

```yaml
- name: Placeholder frontend directory
  run: mkdir -p app/domain/web/public   # or src/py/app/server/static/web
```

Or factor this into a composite action (see `github-ci.md`).

### Mistake 5: Tracking built assets in git

**Symptom:** Noisy diffs on every PR, merge conflicts in minified JS, reproducibility concerns.

**Fix:** Add the bundle dir to `.gitignore`:

```gitignore
# https://github.com/litestar-org/litestar-fullstack-inertia — .gitignore
app/domain/web/public/*
app/domain/web/public/hot
!app/domain/web/public/.gitkeep
.vite
resources/lib/generated/*
!resources/lib/generated/page-props.user.ts
```

Keep the directory itself with `.gitkeep`, gitignore everything else. CI rebuilds per run.

## Summary

1. Vite's `outDir` (or litestar-vite's `bundle_dir`) → inside a Python package.
2. Pick `force-include` OR `ignore-vcs = true` in `[tool.hatch.build.*]`.
3. Make `build-wheel` depend on `build-assets` in the Makefile.
4. `uv build --wheel` runs last, captures everything.
5. Gitignore the build outputs; CI rebuilds per run.
